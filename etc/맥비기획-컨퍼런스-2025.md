### AI 스킬

#### 바이브 기획 방법론 - AI잡돌이, 윤태섭

- 기획의 틀 변경: PPT(사람 중심) -> Markdown(AI 중심) 형태의 파일로 변경
- 기획 산출물의 형태가 바뀌어야 한다.
  - => 공감하긴 함.
  - => 어떻게?
- 보통의 개발자들: 만들줄은 아는데 뭘 만들어야 할 지 모름.
  - 과거에는 '개발 능력' 자체가 희소 자원이었으나, 이제는 대체 가능.
  - 현재는 '무엇이 필요한지 발견하는 능력'이 진짜 희소한 자원
- 산출물 형태의 변경 압력
  - 이제는 사람이 기획을 보고 개발하는게 아님
  - AI가 기획을 읽고 개발: MD양식이 적절. 데이터라면 CSV, JSON 등의 형식이 적절.
- 그러나 현실적으로 아직 과도기이므로, 둘 다를 잡아야 한다.
  - => 근데 기획자한테 다짜고짜 터미널을 들이밀면..???? 클로드 코드 좋은건 인정하는데...
- 자기가 편한 도구를 쓰는게 정답이긴 함. 다만 모델은 적당히 용도별로 잘 골라서.

- 일반적인 기획 플로우
  - 아이디어 -> 컨셉 -> 기능 -> 고객여정 -> 세일즈 -> MVP -> 개발 착수
- 클로드 코드로 이 과정을 따르면 단계별로 산출물을 자동으로 생성됨.
- 사용시에 CLAUDE.md등으로 지침을 줄 수 있음
  - => 우선 순위 설정이 매우 중요함.. 지침만 무작정 많이 주면 AI는 충돌을 해소하지 못함.

- 참고자료: [Appkit GitHub](https://github.com/rupy1014/appkit)
  - claude command 예제

- AI가 뽑아주는 내용은 사실상 초안. 5~60% 수준. 손질이 필요함.
  - 중요한건 이터레이션: 점진적으로 구체화/정제하는 과정이 필요함.

- MD로 초안 잡고, MCP, Skills를 통해 노션, PPT, pdf, Jira등으로 변환

- 순서가 중요하다
  - 산출물보다는 그 내용이 중요함.
  - md 파일이 AI가 다루기 편리함. 이를 통해 정수를 만들고 이를 추가 가공하는 순서가 효율적

- => 예시로 소설 작성 AI 에이전트 그룹을 들었는데.. 여기도 두개 축으로 구성이 필요하지 않을까 싶네
  - => 할루시네이션 방지를 위해 서브 에이전트들을 통해 검토하는것도 좋은듯.
  - => 그러나 이들도 각자 별개의 문맥을 가질테니.. 통일된 큰 문맥을 관리하기 위한 마스터 문맥을 두어야 하고.. 이건 영속적으로 유지해야 하니까 실제로 거대한 '사실 묘사' 파일이 하나 있어야겠네
  - => 생각해보니 이게 spec-kit네?
